# Pipeline Lessons

- 2026-04-13: CLAUDE.md should stay under 100 lines. Domain-specific instructions belong in skills loaded on demand, not in the root file.
- 2026-04-13: Slash commands using `context: fork` run in isolated context windows — their length doesn't stack onto the main session. But they do load CLAUDE.md plus their own instructions, so total context at fork start matters.
- 2026-04-13: MCP can't be tested from bash hooks — it lives inside Claude Code's runtime. Use a lightweight SOQL probe (`SELECT Id FROM Organization LIMIT 1`) as the first step of any slash command that depends on MCP.
- 2026-04-13: `say` command speaks out loud through active audio output. Never use in distributed tooling — use `osascript` notifications only.
- 2026-04-13: `.claude/agents/` defines subagents for CLI-driven sessions (`claude --agent <name>`), not for VS Code extension chat. The `model:` frontmatter in commands/skills only applies to the initial invocation, not subsequent messages. For model persistence in VS Code, use `/model <alias>` after invoking the command. The `/model` switch preserves full conversation history.
- 2026-04-14: ~~WebFetch and WebSearch are Claude Code tools available to any command — they just need to be listed in `allowed-tools`. They're especially valuable for Opus commands (sparring, architect) where the context window can absorb the extra tokens and the model can judge when a lookup is worth the cost. Sonnet execution commands generally don't need them.~~ Superseded — see below.
- 2026-04-14: WebFetch and WebSearch don't work on AWS Bedrock. WebFetch uses Haiku internally for content processing; Bedrock doesn't support on-demand Haiku invocations. Both tools were added and removed in the same day. If the provider ever changes to Anthropic API direct, re-add them — the integration worked at the tool level, only the infrastructure blocked it. **Updated 2026-04-15:** Tested via sub-agents (Haiku model, claude-code-guide type) — same failure. The Haiku sub-processor is baked into WebFetch itself, not the calling context. HTTP requests succeed (network layer works), but content processing fails with `400 Invocation of model ID anthropic.claude-haiku-4-5-20251001-v1:0 with on-demand throughput isn't supported`. No workaround exists on Bedrock. The `claude-code-guide` agent appears to work because it reads bundled local documentation via Glob/Grep/Read, not via web access.
- 2026-04-14: Auto mode (`--enable-auto-mode`) requires Anthropic API direct — not available on Bedrock, Vertex, or Foundry. The classifier runs server-side on Anthropic infrastructure. For Bedrock users, `bypassPermissions` is the closest alternative for reducing permission fatigue. Monitor Anthropic's release notes for Bedrock auto mode support.
- 2026-04-14: `bypassPermissions` (formerly `--dangerously-skip-permissions`) is acceptable for demo org work where blast radius is low. The SE confirmation gates in deployment-rules are prompt-level, not permission-level — they remain enforced regardless of permission mode. This is the key safety layer for scout-building.
- 2026-04-14: Prefer `permissions.allow` rules over `bypassPermissions` mode. Allow rules are persistent, targeted, and don't require the SE to remember a startup flag. They also survive session restarts. For this pipeline, pre-allowing all MCP tools + common Bash patterns eliminates prompts without disabling the permission system entirely. Put them in `.claude/settings.json` (committed, distributes with repo), not `settings.local.json` (gitignored, local only).
- 2026-04-14: Bedrock blocks all Anthropic-hosted platform features: auto mode classifier, WebFetch/WebSearch Haiku sub-processor. The model itself works fine — it's the surrounding infrastructure that's missing. When evaluating new Claude Code features, check the provider requirements first. Monitor Anthropic release notes for Bedrock auto mode support — when it ships, it replaces `permissions.allow` as the primary permission strategy.
- 2026-04-14: CLAUDE.md must not contain per-user editable values (org alias, username, etc.) — distributed users get merge conflicts on every `git pull`. Org identity lives in `sf config`; session-startup.sh bridges it into conversation context dynamically. Committed files must be stable templates that never change per-user.
- 2026-04-14: Critical user actions (model switching, org confirmation) must be standalone blocking gates with osascript notifications — not lines embedded in multi-item output blocks. New users in VS Code are overwhelmed and will skip the third item in a three-item message every time.
- 2026-04-14: Final bookkeeping steps (LOGs, change logs, lessons) must complete before the "done" signal to the SE. Sonnet treats SE interaction as a natural stopping point — any step that comes after the message saying "complete" or "done" will be skipped. Structure commands so the completion notification is always the last thing that fires, with all logging and bookkeeping as preceding substeps.
- 2026-04-14: `permissions.allow` patterns for Bash match against the full command string using glob syntax. `Bash(sf config:*)` is wrong — the actual command uses spaces, not colons: `Bash(sf config *)`. Colon-separated patterns silently fail to match, causing unexpected permission prompts with no error message.
- 2026-04-15: VS Code throws errors on `.claude/skills/` folder names with leading underscores. The `_demo-*` convention was chosen to visually separate internal skills from community skills, but it causes tooling issues. Use a plain prefix (`demo-`, `pipeline-`) instead — still visually distinct, no VS Code complaints.
- 2026-04-15: Sonnet's "natural stopping point" problem extends beyond the done-signal pattern (documented 2026-04-14). Even within a single step, if verification produces terminal output before the LOG is written, Sonnet treats it as complete. The fix is structural: sub-steps (verify, log, report) must be explicitly sequenced within a single numbered step with "do not output anything until all sub-steps are complete." Separate numbered steps with SE-visible output between them will always be vulnerable to early termination.
- 2026-04-16: Sub-agents (Agent tool) inherit MCP tool access. Verified: a Sonnet sub-agent can call run_soql_query, and by extension deploy_metadata, retrieve_metadata, assign_permission_set — the full Salesforce DX MCP suite. No workaround needed. This enables phased deployment architectures where each sub-agent gets a clean context window with only the skills it needs.
- 2026-04-16: Opus orchestrator + Sonnet executors is the right model split for deployment commands. The orchestrator needs judgment (prompt construction from spec + skills, failure interpretation, change log synthesis). The executors need speed and instruction-following (metadata XML generation, sequential MCP calls). Use the Agent tool's `model` parameter to enforce this: orchestrator runs as Opus, sub-agents as Sonnet.
- 2026-04-16: Community skill libraries (forcedotcom/afv-library) follow the same Agent Skills spec (SKILL.md with YAML frontmatter) we use. Skills can be adopted by copying SKILL.md — no structural adaptation needed. Specialized depth often beats generalist breadth for code generation tasks. Evaluate periodically; the library is actively maintained.
- 2026-04-16: Before adopting MCP-dependent community skills, verify the required MCP tools are available in our .mcp.json toolsets. AFV's generating-flow requires `execute_metadata_action` which is not in our current server config. The skill itself is excellent but unusable without the tool dependency.
- 2026-04-16: Retry/debugging work from failed deployments is inherently investigative — diagnosing managed package blockers, trying workarounds, adapting in real-time. This is orchestrator-level reasoning, not templated sub-agent work. If the SE invokes `/scout-building` with a change log retry request, the orchestrator handling it directly is acceptable behavior, not a bug. Revisit if retry patterns become common enough to template.
- 2026-04-17: Model version numbers ("Opus 4.6", "Sonnet 4.6") do not belong in SE-facing copy. They rot every ~60 days as new versions ship. Use the alias only ("Opus", "Sonnet") — the `model:` frontmatter resolves to the latest automatically, so user-facing text must be version-agnostic. Versions belong in release notes and commit messages, nowhere else.
- 2026-04-17: Sub-agents inherit Skill tool access from the parent session. Do not inject full skill file contents as `{{placeholder}}` strings in sub-agent prompts — it wastes ~10-20k tokens in the orchestrator's context per run. Tell the sub-agent which skills to invoke by name; let it Skill-call them itself. This also keeps the prompt templates short and maintainable.
- 2026-04-17: Prompt templates are not skills. If a file is never description-matched and never grants a discoverable capability — if it's just a string template with placeholders that one specific step reads and fills — it belongs in `.claude/prompts/`, not `.claude/skills/`. Using SKILL.md as a prompt template forces awkward "strip the frontmatter" workarounds that were documented then retired in the same week.
- 2026-04-17: For structured sub-agent output, require a fenced JSON block rather than literal section headers. Header-matching ("DEPLOYED:", "SKIPPED:") is brittle — one rogue rephrase by Sonnet triggers a false-negative validation failure. JSON gives the orchestrator real parsing and real schema enforcement.
- 2026-04-17: Blocking text gates in VS Code do not need osascript notifications. The markdown blockquote is already prominent in the chat surface. Notifications make sense for background handoffs (sub-agent spawned, SE gate needed mid-autonomous-run) — not for every inline confirmation. Trim the ceremony where the SE is already looking at the chat.
- 2026-04-17: Declarative skill manifest (`.claude/skills-manifest.yaml`) is the right abstraction for multi-source external skill sets. Before this, install.sh hard-coded three separate bash loops (Jaganpro/AFV/ADLC) and removal required editing code. With a manifest: install.sh, /sync-skills, and session-startup drift detection all read the same file — add/remove a skill = one YAML edit. Orphan pruning, upstream-vs-synced SHA drift detection, and install.sh-changed detection become cheap. Homegrown skills (`demo-*`, `pipeline-lessons`) stay out of the manifest — they ship with the repo and move via `git pull`. pyyaml is a hard dependency, installed by install.sh. This pattern generalises: anywhere the pipeline hard-codes a list of external artefacts across a loop, a manifest is likely warranted.
- 2026-04-17: `sf config set target-org` has local vs global scope. Local (`.sf/config.json` in the project) always beats global (`~/.sf/config.json`). MCP reads whatever the CLI resolves, so a stale local file will silently shadow a correct global one — CMD+Q can't fix it. Always write to local for project-scoped orgs. **Why:** today's session — /switch-org wrote global, local was stale with a deleted alias, MCP couldn't connect and the failure mode looked like "MCP bug" when it was actually config shadowing.
- 2026-04-17: Slash commands are auto-indexed into the skills menu by the harness, even when no SKILL.md exists. In forked-context commands, Sonnet/Opus can see its own command name listed as a "skill," hunt for a missing SKILL.md, and bail. Add a preamble disclaimer to every `context: fork` command that tells the model "ignore the menu entry, your instructions are this file." **Why:** project-building consistently stalled at Step 1 with "the project-building skill doesn't exist" — same root cause, defensive fix applied to scout-building too.
- 2026-04-18: Remote HTTP MCP servers bypass the Bedrock Haiku sub-processor block that killed WebFetch/WebSearch. The Salesforce Docs MCP (`salesforce-docs-76258744c9d7.herokuapp.com/api/mcp`) is a plain HTTP server — Scout's MCP client talks HTTP to it, Bedrock is out of the loop for the fetch itself. This is the pattern for any future doc/reference lookup need on Bedrock. Different servers are namespaced by tool prefix (Salesforce DX exposes `run_soql_query`/`deploy_metadata`/etc.; Docs MCP exposes `salesforce_docs_search`/`salesforce_docs_fetch`) so they coexist without conflict. **Why:** every April 14-15 lesson about web-tool removal was scar tissue from Bedrock's Haiku block on Anthropic-hosted tools. Remote MCP is the workaround.
- 2026-04-18: Docs consultation is targeted, not ambient — every Salesforce Docs MCP call must earn its latency. Consult only for release-gated features, novel metadata types, unfamiliar deploy errors (pre-retry), SE-referenced unknowns, and non-trivial Agent Script elements. Do NOT consult for standard object structure, custom field/object/permset XML (afv-library skills are authoritative), customer-asserted facts, or anything consulted already this session. Decision tree lives in `demo-docs-consultation` skill. Every call produces a citation (URL + question + verdict) that lands in the spec (sparring) or change log (building). **Why:** ambient verification tax would slow Scout without improving quality on 90% of calls — org audit and skill-library rules already cover the common case. Docs earn their keep on the edges: release accuracy, error diagnosis, Agent Script capability drift.
  - 2026-04-18: Orchestrator-level rules don't reach sub-agents mid-loop. The 1914 PLAN put "consult docs before retry on unfamiliar error" in scout-building.md, but Phase 1/2 sub-agents run their full fail-twice-then-SKIP loop inside a single invocation — the orchestrator only sees the final SKIP, not the first failure. Error-recovery instructions must live in the sub-agent's own Deployment Rules to fire between attempt 1 and attempt 2. **Why:** caught during critical evaluation of 1914 — listing a skill in "Skills Available" doesn't trigger it; the trigger must live at the decision point in the instructions the sub-agent actually reads top-to-bottom. How to apply: whenever orchestrator-level behaviour depends on sub-agent state, the rule must be duplicated into the sub-agent prompt template — not referenced from the parent command.
  - 2026-04-18: Scout-specific skills use the `demo-` prefix (demo-deployment-rules, demo-org-audit, demo-docs-consultation). Community skills (sf-flow, sf-apex, generating-*) keep their upstream names. Pipeline-internal skills use `pipeline-` prefix. **Why:** the 1914 PLAN introduced `docs-consultation` without a prefix and drifted from the convention; rename happened 40 minutes later. How to apply: before adding any new skill, check whether it's Scout-owned (prefix `demo-`), pipeline-owned (prefix `pipeline-`), or community (keep upstream name).
    - 2026-04-18: Eager-loaded content that is command-specific belongs in `.claude/prompts/`, not `.claude/skills/`. A skill earns its folder when it (a) holds reusable decision logic, (b) is description-matched by the model, or (c) is read by multiple callers in meaningfully different contexts. Lessons and templates fail all three: they are always-loaded by one command at one step. The `.claude/skills/` folder auto-indexes into the skills menu, which creates noise ("is this a capability I should invoke?") that hurts more than it helps for prompt fragments. **Why:** audit of Scout-native skills (6 total) showed `demo-lessons`, `demo-spec-format`, `demo-change-log` were all eager-loaded command-specific content — relocated to `.claude/prompts/`. Matches the phase1/2/3 precedent from 2026-04-17. How to apply: before adding anything to `.claude/skills/`, ask: does another command or the model's description-matching ever load this? If no, it's a prompt, not a skill.
    - 2026-04-18: `context: fork` is harmful for Opus orchestrators that spawn sub-agents. The auto-indexed skills menu lists the command's own name as a "skill" — with no conversation history to orient from, Opus hunts for a non-existent `scout-building/SKILL.md` despite the defensive disclaimer added 2026-04-17. No other command uses fork. Sub-agents already run in clean contexts by design; the orchestrator benefits from seeing the SE's prior sparring conversation. **Why:** recurring misbehaviour in scout-building where Step 1 stalled on phantom-skill hunts; disclaimer reduced frequency but didn't eliminate. How to apply: do not add `context: fork` to any command that spawns sub-agents. Use fork only for pure one-shot utilities where no inherited context matters — and even then, prefer unforked unless context pollution is demonstrably a problem.
    - 2026-04-18: Sparring → Building handover should cross a fresh Claude Code window. Spec and PLAN files on disk are the contract between the two phases; no conversational state needs to flow through. Running building in the same window inherits sparring's context (~thousands of tokens of discovery, gates, SE back-and-forth) that building will never reference but must still process. **Why:** user-surfaced — Opus sparring sessions grow long, and Sonnet/Opus building inherits that when run in the same window. How to apply: both sparring commands now explicitly instruct the SE to open a fresh window before running the corresponding building command.
    - 2026-04-18: Orchestrator-facing and sub-agent-facing content must not co-habit the same skill file. `demo-deployment-rules` originally mixed gate-firing copy (notification pattern, confirmation question phrasing — orchestrator-only) with per-category deploy procedure (sub-agent-only). Sub-agents reading the skill would hit "fire notification and ask" instructions irrelevant to them; orchestrators loading it got per-deploy procedural noise. Split by audience: gate copy stays inline in scout-building.md (where gates actually fire); per-category procedure stays in the skill (loaded on-demand by phase sub-agents). **Why:** detected while consolidating phase prompts on 2026-04-18 — the skill's two-audience structure was itself the bug that created phase-prompt duplication. How to apply: before writing any pipeline skill, name the single caller audience. If you catch yourself writing "the SE will see..." and "the sub-agent will do..." in the same file, split.
    - 2026-04-19: Opus 4.7 chokes on bulk MCP payloads materially faster than 4.6 did. Scout-sparring's Stage 4 audit (retrieve_metadata across objects + layouts + flows + LWC + agents on SDO-scale orgs) overflowed 4.7's context before it could summarise. The fix mirrors scout-building's sub-agent pattern: delegate bulk ingestion to a Sonnet sub-agent, have it write to disk, return a compact JSON summary to the orchestrator. Opus never sees the raw payloads. **Why:** testing on real demo orgs (290+ objects, 380+ flows, 499+ LWC) made the direct-Opus audit path unusable after the 4.6→4.7 upgrade. How to apply: any new Opus command step that would call retrieve_metadata across unbounded result sets (all objects, all flows, all layouts) must be delegated to a Sonnet sub-agent with a disk-output + JSON-summary contract. Direct Opus MCP calls are acceptable only for bounded single-record reads (one layout, one flow, one record count).
    - 2026-04-19: Sub-agent output must be spot-checked, not blindly trusted. When Opus delegates bulk work to Sonnet and receives a compact JSON summary, the summary is only as good as the sub-agent's discovery methods. Two defenses: (1) explicit discovery instructions per section in the prompt template — don't leave Sonnet to guess which API or SOQL table to hit, because it will guess wrong on less-common metadata types (Agentforce, GenAiPlanner). (2) Opus runs a cheap parallel spot-check (3 count queries) after receiving the summary, catching gross mismatches before they poison the spec. **Why:** scout-sparring reported "no agents" in an org with 7 bots because the audit template said "list agents" without saying how, and Opus had no verification step. **How to apply:** any future sub-agent that writes a summary for Opus must have explicit discovery methods per section AND the orchestrator must spot-check at least the sections most likely to silently fail.
    - 2026-04-19: Sub-agent prompts that say "retrieve in batches" without a completion gate will silently truncate. The fix is count-first, enumerate-selectively: run a SOQL COUNT() first (this is what the orchestrator spot-checks against), then enumerate only the objects the demo will touch. This also applies to LWC and any other unbounded metadata type. Pair with a pre-return completeness checklist so the sub-agent self-verifies before emitting its JSON summary. **Why:** audit sub-agent enumerated 80 of 720 active flows and reported PARTIAL with no indication of truncation scope. **How to apply:** any sub-agent prompt that retrieves unbounded metadata sets must include (1) a count-first step, (2) selective enumeration criteria, and (3) a checklist gate before the output block.
    - 2026-04-19: Hook-script banner output feeds the model's opening context via system-reminder. When a shell script parses CLI JSON with brittle regex (`"key":"value"` without whitespace tolerance), pretty-printed JSON silently returns empty and the banner lies to the model. The model then trusts the banner over its own tools and bails. **Why:** today — session-startup.sh said "no default org set" while `sf config get target-org --json` clearly returned `"value": "demo-org"`; Opus refused to run the MCP probe at scout-sparring Stage 1 and told the SE to run /switch-org. Derailed the whole session. How to apply: (1) any hook-script JSON parse must tolerate `[[:space:]]*` around the colon, or use `python3 -c 'import json'` / `jq` for anything non-trivial — raw grep on JSON is a bug waiting to happen. (2) Commands that probe their own environment (MCP, auth, org connectivity) must treat the probe as ground truth over banner signals, because banners run before the session and can report stale or miscomputed state. Scout-sparring Stage 1 updated accordingly.
    - 2026-04-19: Community skills expand the automation boundary over time. When new skills are adopted (sf-data, testing-agentforce, generating-custom-field), re-evaluate what the SE Manual Checklist claims is "always manual" — picklist values, queues, agent smoke tests, and data seeding patterns may have moved into automatable territory. **Why:** review of installed skills against hardcoded SE Manual items found 4 items that were automatable but still listed as manual, because the checklist was written before those skills existed. **How to apply:** do this review periodically (e.g., after each /sync-skills adoption wave), not just when a specific request surfaces the gap.
    - 2026-04-19: Tooling API `Layout.TableEnumOrId` stores the entity key ID (e.g. `01IgL000005ExOjUAK`) for custom objects, but the string API name (e.g. `Account`) for standard objects. `WHERE TableEnumOrId = 'MyObject__c'` silently returns 0 for all custom objects. Use `WHERE Name LIKE '[Label]%Layout%'` as the fallback for custom objects without ProfileLayout entries. **Why:** golden-vs-subagent audit comparison showed both audits failing to find Makana_Part__c and Makana_Van_Inventory__c layouts. The Tooling query looked correct but returned 0 because of the key-ID-vs-string asymmetry. **How to apply:** any audit or deployment code that queries Tooling API `Layout` for custom objects must use `Name LIKE` pattern, not `TableEnumOrId = '[ApiName]'`. Standard objects can continue using `TableEnumOrId` directly.
    - 2026-04-19: Salesforce Docs MCP works for architectural research, not just error recovery. The case-routing question surfaced two viable approaches (Einstein Case Classification vs. Agentforce agent-from-flow) with specific setup steps and constraints — all grounded in official docs with citations. **Why:** the original docs-consultation decision tree was scoped narrowly to release-gated feature verification and deploy-error diagnosis. Architectural "how should we build this?" questions are a natural third use case that the same MCP handles well. **How to apply:** during sparring, when the SE proposes a solution architecture (not just a feature), consult docs to validate the approach and surface alternatives the SE may not know about. This is especially valuable for Agentforce patterns where the platform ships new invocation methods frequently.
    - 2026-04-19: A single Sonnet sub-agent cannot audit an SDO-scale org — 100+ tool calls produce output that exceeds the 8K output token cap. Split by domain (standard objects / apps+flows+agents+LWC / custom objects+permsets) keeps each agent under 50 tool calls. The fragments concatenate via Bash; Opus writes the narrative (Notable Gaps) from JSON summaries only, never reading the raw markdown. Both the standard-objects and apps/flows/agents sub-agents independently discover the default app — the orchestrator spot-checks both against its own UserAppInfo query to break ties. **Why:** lsdo-2604 stress test (290+ objects, 723 active flows, 14 bots, 499+ LWC). **How to apply:** any future sub-agent prompt that would produce 80+ tool calls on a large org should be evaluated for domain splitting. The 8K cap is a hard wall, not a quality knob.
    - 2026-04-19: Parallel deployment sub-agents don't work — Salesforce Metadata API allows only one active deployment per org at a time. Concurrent `deploy_metadata` calls return `ALREADY_IN_PROGRESS`. Parallel reads (retrieve_metadata, run_soql_query) are fine — that's why audit splitting works. Additionally, AIAuthoringBundle has new per-type 24-hour deployment volume limits (Summer '25 release notes) that could affect Agentforce retry loops. **Why:** evaluated parallel building sub-agents for speed; rejected after docs confirmation. **How to apply:** never split deployment phases into parallel sub-agents. Sequential phase execution (Phase 1 → 2 → 3) is structurally required, not just a design choice.
    - 2026-04-19: Docs consultation is a design input, not a feasibility gate. For industry clouds (LSC, FSC, Insurance), the platform data model is too rich to guess correctly — standard objects like HealthcareProvider, Inquiry, and MedicalInsight are invisible without docs. But even standard Sales/Service orgs benefit: Agentforce ships features monthly, Flow capabilities evolve, and Data Cloud patterns change. Move doc research before scenario proposal, not after. The latency cost is 3-7 searches; the quality payoff is a scenario grounded in current platform reality instead of stale training data. **Why:** scout-sparring proposed `Adverse_Event_Report__c` custom object and KOL fields on Contact for an LSC org — docs revealed HealthcareProvider (84 records), Inquiry with LSDO_Adverse_Event record type, BoardCertification, and MedicalInsight were all already standard objects in the org. Entire scenario data model had to be rebuilt. **How to apply:** docs consultation runs unconditionally before scenario proposal (Stage 6), not after (old Stage 7). SE names the industry cloud in Stage 5; docs search uses that to find the data model.
    - 2026-04-19: Don't hardcode domain-specific lists when the SE is a better signal. Salesforce has 16+ industry clouds (Health Cloud, LSC, FSC, Manufacturing, Automotive, Consumer Goods, Insurance, Education, Government, Nonprofit, Net Zero, Communications, Media, Energy & Utilities, Retail Execution, Philanthropy) — each with its own standard objects. A curated list will always lag reality and rot silently. Instead: (1) the SE names the cloud (they're paid to know this), (2) docs search finds the data model, (3) the audit provides a lightweight safety net by surfacing non-universal standard objects with data in `demo_surface_notes`. The SE drives; the tooling confirms. **Why:** initial 1739 PLAN hardcoded 20 industry objects — immediately incomplete (missed Automotive, Consumer Goods, Education, etc.) and would need updating every release. **How to apply:** when tempted to hardcode a Salesforce domain taxonomy, ask whether the SE already knows the answer. If yes, ask them and verify with docs/audit. Hardcode only universal invariants (Account, Contact, Case — these never change).
